# 数字孪生水网平台总体任务清单

## 1. 技术路线

- [ ] 前端采用 Vue + TypeScript + Vite，负责页面交互、地图展示、图表展示和仿真配置。
- [ ] 后端采用 Spring Boot，负责 REST API、权限、业务逻辑、模型调用、数据持久化。
- [ ] 主底图采用 2D 天地图，承载常规业务操作和水网监控。
- [ ] 辅助视图采用 Cesium 3D，承载三维预演、水位变化、水面渲染和演示亮点。
- [ ] 水动力模型由模型组提供，平台侧负责通过约定接口对接、运行调度、结果入库与可视化。
- [ ] 数据库优先采用 MySQL；开发阶段可保留 H2/SQLite 作为轻量替代方案。

## 1.1 界面风格与布局偏好

- [ ] 整体采用水利工程指挥舱风格，深色科技底色，青蓝色突出水网、模型和实时数据。
- [ ] 顶部设置业务模块导航栏，用于切换数据底板、实时监测、仿真预演、风险预警、预案管理、系统管理等模块。
- [ ] 首页采用中间主视图 + 左右数据面板布局：中间展示 2D 天地图或 Cesium 3D 场景，两侧展示指标、图表、告警、工况和监测数据。
- [ ] 两侧面板保持信息密度高但层级清晰，避免堆叠过多装饰性卡片。
- [ ] 图表以水位、流量、流速、降雨、预警等级、模型结果对比为主，服务业务研判。
- [ ] 重要状态采用颜色编码：正常为青绿，关注为黄色，预警为橙色，危险为红色。
- [ ] 2D/3D 视图支持在主视图区域内切换，不改变顶部导航和两侧业务面板结构。
- [ ] 普通业务页面沿用同一视觉体系，但可降低大屏装饰感，优先保证表单、查询、详情和图表可用性。

## 3. 后端搭建任务

- [ ] 调整 Spring Boot 项目基础配置：端口、跨域、JSON 时间格式、统一异常处理。
- [ ] 建立标准包结构：`config`、`common`、`auth`、`network`、`realtime`、`simulation`、`warning`、`scenario`、`history`。

## 4. 数据库设计任务

- [ ] 设计河段表 `river_segment`：编码、名称、长度、宽度、糙率、起止节点、几何线。
- [ ] 设计节点表 `water_node`：编码、名称、类型、经纬度、初始水位、边界类型。
- [ ] 设计水利工程表 `hydraulic_structure`：闸、泵站、规模、控制参数、所属节点。
- [ ] 设计测站表 `monitor_station`：站点类型、位置、关联节点、数据来源。
- [ ] 设计实时/历史监测表 `sensor_record`：水位、流量、流速、水质、采集时间。
- [ ] 设计气象记录表 `weather_record`：降雨、气温、风速、预报时间。
- [ ] 设计仿真工况表 `simulation_scenario`：名称、边界条件、参数、创建人。
- [ ] 设计仿真运行表 `simulation_run`：运行状态、开始时间、结束时间、输入输出文件路径。
- [ ] 设计仿真结果表 `simulation_result_series`：节点/河段、时间步、水位、流量、流速。
- [ ] 设计预警规则表 `warning_rule`：指标、阈值、级别、适用对象。
- [ ] 设计告警事件表 `warning_event`：触发时间、指标值、处理状态、确认人。
- [ ] 设计预案表 `dispatch_plan`：调度措施、适用场景、效果评价。
- [ ] 设计用户、角色、权限相关表。
- [ ] 绘制 ER 图并补充字段说明、主键、外键和索引。

## 5. 水网基础数据任务

- [ ] 根据 PPT 提供的范围建立初始河网：蠡河、大溪港、京杭运河及试点片区核心河段。
- [ ] 内置示例河段参数：
  - [ ] `Lihe_S1`：L=1.4km，B=20m，n=0.03。
  - [ ] `Lihe_S2`：L=6.4km，B=20m，n=0.03。
  - [ ] `Jinghang_S1`：L=1.5km，B=105m，n=0.03。
  - [ ] `Jinghang_S2`：L=6.1km，B=105m，n=0.02。
  - [ ] `Jinghang_S3`：L=1.5km，B=105m，n=0.02。
  - [ ] `Daxigang`：L=5.2km，B=30m，n=0.03。
- [ ] 建立边界条件示例：上游流量 Q=20m3/s，下游水位 Z=2.5m。
- [ ] 补充闸站、泵站、监测点的示例数据。
- [ ] 确定断头河处理方式和水网概化说明，作为报告评分点材料。

## 6. 外部水动力模型对接任务

- [ ] 与模型组确认模型交付形式：Python 脚本、可执行程序、HTTP 服务或容器服务。
- [ ] 与模型组确认模型启动方式：命令行参数、配置文件路径、工作目录、运行环境和依赖。
- [ ] 与模型组共同约定模型输入 JSON 格式：河段、节点、边界条件、时间步长、计算步数、糙率、河宽。
- [ ] 与模型组共同约定模型输出 JSON/CSV 格式：水位、流量、流速、水面线、运行摘要。
- [ ] 平台侧提供示例输入文件和接口说明，供模型组按格式适配。
- [ ] 平台侧先实现模型适配器 `ModelRunner`，屏蔽具体模型实现方式。
- [ ] 平台侧提供 Mock 模型适配器，用模拟结果打通前后端演示链路。
- [ ] 平台侧实现真实模型适配器，在模型组交付后切换到真实求解器。
- [ ] 平台侧捕获模型运行日志、标准输出、错误输出和退出码。
- [ ] 平台侧维护模型运行状态：排队中、运行中、成功、失败、超时、已取消。
- [ ] 平台侧保存每次计算的输入文件、输出文件、运行日志和结果摘要。
- [ ] 平台侧负责解析模型结果并写入仿真结果表。
- [ ] 平台侧负责把结果转换为前端图表、地图图层和 Cesium 动态播放所需的数据结构。
- [ ] 平台侧不负责水动力核心算法开发，只负责接口对接、任务调度和结果展示。

## 7. 仿真接口任务

- [ ] `POST /api/simulations/run`：发起仿真计算。
- [ ] `GET /api/simulations`：分页查询历史仿真记录。
- [ ] `GET /api/simulations/{id}`：查询仿真详情。
- [ ] `GET /api/simulations/{id}/series`：查询水位、流量、流速时间序列。
- [ ] `GET /api/simulations/{id}/profile`：查询某时刻水面线。
- [ ] `POST /api/simulations/{id}/compare`：对比不同工况结果。
- [ ] 为核心接口编写接口测试。

## 11. 四预业务任务

- [ ] 预报：基于边界条件和模型计算输出未来水位/流量变化。
- [ ] 预警：配置水位、流量、水质阈值，超限自动生成告警。
- [ ] 预演：配置降雨、太湖水位突变或上游来水工况，播放模拟过程。
- [ ] 预案：配置闸门/泵站调度方案，对比不同方案的减灾效果。
- [ ] 生成风险热力图或风险等级图层，用于演示。

## 12. 水环境业务任务

- [ ] 根据模型结果识别低流速、弱交换河段。
- [ ] 建立水动力弱河段列表和地图高亮。
- [ ] 设计至少一种改善措施，例如开闸引排、泵站调水、提高循环。
- [ ] 通过仿真对比改善前后的流速、水位或交换能力。
- [ ] 输出水环境分析结论，用于报告和演示。
